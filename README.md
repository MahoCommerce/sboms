# Maho SBOMs

[![Refresh](https://github.com/MahoCommerce/sboms/actions/workflows/refresh.yml/badge.svg)](https://github.com/MahoCommerce/sboms/actions/workflows/refresh.yml)
[![Critical](https://img.shields.io/endpoint?url=https://raw.githubusercontent.com/MahoCommerce/sboms/main/badges/critical.json)](VULNERABILITIES.md)
[![High](https://img.shields.io/endpoint?url=https://raw.githubusercontent.com/MahoCommerce/sboms/main/badges/high.json)](VULNERABILITIES.md)

Software Bill of Materials (SBOM) inventory for the [mahocommerce](https://github.com/mahocommerce) organization.

This repository tracks one CycloneDX SBOM per repo per ref (HEAD of the default branch and the latest release tag), refreshed daily by GitHub Actions. It exists so users, integrators, and security auditors have a single, version-controlled place to see what's inside every Maho project.

## Why one SBOM per repo (not one combined file)

An SBOM document, by definition, describes the components of *one* piece of software. Its `metadata.component` field names the subject. Combining unrelated products (e.g. `maho` and `maho-dashboard`) into a single document would be semantically wrong: scanners would treat the union as a single deployable, and you'd lose the ability to attribute a vulnerability to a specific product. So we keep one SBOM per product, in a single repo.

## Layout

```
bin/                      # generation & scanning tooling
  refresh.sh  scan.sh     # orchestration (entry points)
  inject_license.py  merge.py  report.py
sboms/
  <repo>/
    main.cdx.json         # HEAD of the default branch
    <release-tag>.cdx.json  # latest GitHub release, when one exists
licenses/
  <repo>/
    main.json             # merged license summary of the repo's own source
    <release-tag>.json
vulns/
  <repo>/
    main.json             # merged Grype + Trivy findings for HEAD of default branch
badges/                   # shields.io endpoint JSON (critical/high counts)
```

`sboms/*.cdx.json` are CycloneDX 1.6 JSON, generated by [Syft](https://github.com/anchore/syft) — a **dependency**-level inventory (which third-party components ship in each product, and their licenses).

`licenses/*.json` summarize the **source**-level licensing of each repo's own files. [REUSE](https://reuse.software/) reads the per-file `SPDX-License-Identifier` headers (introduced in maho#939) via `reuse lint --json`; a `jq` filter in `bin/refresh.sh` keeps REUSE's native `summary` object verbatim and adds a per-license file count, condensing REUSE's multi-megabyte per-file report into a few hundred bytes:

```json
{ "repo": "maho", "ref": "main", "reuse_spec_version": "3.3",
  "summary": { "used_licenses": ["AFL-3.0", "MIT", "OSL-3.0"],
    "files_total": 5951, "files_with_copyright_info": 5625,
    "files_with_licensing_info": 5622, "compliant": false },
  "license_file_counts": { "AFL-3.0": 1136, "MIT": 3, "OSL-3.0": 4484 } }
```

The `summary` block is REUSE's own schema, so the report stays recognizable to REUSE-aware tooling. The two artifacts are complementary: CycloneDX answers "what's bundled," this answers "how is our own code licensed." It's deterministic (no volatile document UUID/timestamp), so it only changes when licensing actually changes.

Each CycloneDX SBOM's top-level component also carries the project's declared license: Syft only resolves dependency licenses, so `bin/refresh.sh` injects `metadata.component.licenses` from the repo's `composer.json` / `package.json` (`bin/inject_license.py`).

## How it stays current

A single scheduled workflow (`.github/workflows/refresh.yml`) runs daily at 03:00 UTC (or on demand via `workflow_dispatch`). It has two jobs that run in sequence: `refresh` regenerates the SBOMs, then `scan` re-scans them for vulnerabilities. Scanning depends on the refresh so it always runs against the freshest SBOMs, and both share the `sbom` concurrency group so runs never overlap.

### `refresh` job

1. Runs `./bin/refresh.sh`, which enumerates public, non-archived, non-fork mahocommerce repos via the GitHub API. `maho-language-*` repos are skipped (they are auto-generated splits of the `maho-l10n` monorepo and contain no first-party dependencies), as is `sboms` itself (this repo — self-referential, not a tracked product).
2. For each repo, clones the default branch and every published release tag, then runs Syft. Release-tag SBOMs already on disk are skipped (tags are immutable), so the first run backfills the full history and subsequent runs only generate new tags plus refreshed `main`.
3. Injects the project's declared license into each CycloneDX SBOM's main component (`bin/inject_license.py`) and writes a merged source-license summary from the clone's per-file headers (`reuse lint --json` piped through a `jq` filter).
4. Commits any changes under `sboms/` and `licenses/` back to this repo.

To exclude a repo, archive it on GitHub or make it private. Both states are filtered out automatically.

### `scan` job

The job first fast-forwards to the branch head (the `refresh` job may have just pushed new SBOMs), then runs `./bin/scan.sh`. For each `sboms/<repo>/main.cdx.json`:

1. Runs **Grype** (`--only-fixed`) and **Trivy** (`--ignore-unfixed`). Both tools skip vulns with no upstream fix available, which removes most of the noise.
2. Merges the two outputs by `(cve, package, version)` via `bin/merge.py`, picking the higher severity when scanners disagree and recording which tool(s) flagged each finding.
3. Writes results to `vulns/<repo>/main.json`.

After every repo is scanned, `bin/report.py` aggregates the findings into a top-level [`VULNERABILITIES.md`](VULNERABILITIES.md) — a summary table plus a Critical/High detail section — and writes the shields.io badge endpoints under `badges/`. The same report is published to the workflow run's job summary, so you can read it directly in the Actions tab. The job then commits any changes under `vulns/`, `VULNERABILITIES.md`, and `badges/`, and finally fails the run if any merged finding is **Critical**, so the data is always current and the failure status is the actionable signal.

Old release tags are not scanned: they are immutable history and alerting on them helps no one. If a finding is a known false positive or you've assessed it as not applicable, add a [Grype suppression](https://github.com/anchore/grype#specifying-matches-to-ignore) in `.grype.yaml` at the repo root.

## Consuming the SBOMs directly

```bash
# Vulnerability scan with Grype
grype sbom:./sboms/maho/main.cdx.json

# Or with Trivy
trivy sbom ./sboms/maho/main.cdx.json

# Upload to a Dependency-Track instance
curl -X POST -H "X-Api-Key: $DT_API_KEY" \
     -F "bom=@sboms/maho/main.cdx.json" \
     -F "projectName=maho" -F "projectVersion=main" \
     https://your-dependency-track/api/v1/bom
```
